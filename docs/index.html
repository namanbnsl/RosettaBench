Method

How RosettaBench works

A fully deterministic, LLM-judge-free evaluation pipeline. Each problem gets a unique synthetic language — making memorisation structurally impossible.

01 — Language Generation

Unique per-problem vocabulary

A synthetic language is generated from a per-problem seed using a syllable-based RNG. The remapping covers all Python vocabulary: 35 keywords, 77 builtins, 55 methods, 25 operators, 4 delimiters. No two problems share a language.

02 — Prompt Construction

6 few-shot example pairs

The model receives 6 in-context examples pairing synthetic and Python code, followed by the problem statement. It must respond with a <solution> block written entirely in the synthetic language.

03 — Extraction & Validation

Token-level leak detection

Responses are checked for Python token leaks (PYTHON_LEAK). If clean, the code is back-translated to Python for execution. This makes the validation reproducible and removes any subjectivity from judging.

04 — Scoring

All-or-nothing correctness

A problem is correct only if all test cases pass. Final score is passed / total. Learning Tax = baseline − core. No LLM judge is used at any stage.

Key Insights

What the data reveals

Hard problems

The most discriminating tier

11 of 16 models score under 10% on hard problems. Hard problems require language acquisition and algorithmic ability simultaneously — models cannot compensate for language failure with pattern-matching alone.

Failure modes

Memorisers revert to Python

PYTHON_LEAK is the dominant failure mode across 14 of 16 models. Models abandon the synthetic language and fall back on memorised Python tokens — the clearest signal of memorisation over learning.

Gemini exception

Only 2 models truly learn

Gemini 3.1 Pro Preview (7%) and Gemini 3 Flash Preview (10.3%) are the only models with learning tax under 15%. Every other model loses at least 40 percentage points moving from Python to synthetic.

Case study: abc357_b

Memorisers assume, learners adapt

Problem abc357_b required uppercase conversion, but the few-shot examples only demonstrated .lower(). Every model except Gemini 3 Flash and 3.1 Pro hallucinated a .upper() token. The two exceptions used .lower() with a character map, staying within the demonstrated vocabulary.

Conclusion: RosettaBench demonstrates that strong existing benchmark performance does not imply the ability to learn. Measuring this gap is essential for tracking real progress toward AGI. The models didn't fail due to weak coding ability — they failed because they relied on memorisation instead of learning.
